## 1. 项目概述

### 1.1 项目背景
手势识别是人机交互（HCI）的重要方向，广泛应用于智能家居控制、无障碍交互、AR/VR、车载交互、远程演示控制等场景。相比键鼠/触屏，手势交互具备非接触、自然直观的优势。本项目构建一个**基于浏览器的 Web 手势识别系统**：用户只需打开网页、授权摄像头，即可实时识别手势，无需安装任何客户端。系统采用 B/S 架构，浏览器端负责采集与关键点提取，后端负责模型推理与数据/模型管理。

### 1.2 项目目标
- 用户在浏览器中授权摄像头，实时检测手部并识别手势；
- 同时支持**静态手势**（固定姿态）与**动态手势**（连续动作）；
- **原始视频帧不上传**，仅传输 63 维关键点，保护隐私、降低带宽；
- 识别结果可映射到**网页内交互动作**（如翻页、轮播、播放控制、画布绘制）；
- 提供 Web 前端可视化：实时画面、关键点骨架、识别标签、置信度、FPS；
- 提供数据采集页、模型训练触发与模型管理的后台接口；
- 端到端延迟 < 150ms，前端渲染 ≥ 24 FPS，静态手势识别准确率 ≥ 95%。

### 1.3 功能范围（MVP → 进阶）
| 层级 | 功能 | 说明 |
|------|------|------|
| MVP | 浏览器摄像头采集 + 关键点提取 | 前端 MediaPipe Tasks Vision，输出 21 个 3D 关键点 |
| MVP | WebSocket 实时通信 | 前端推关键点，后端回识别结果 |
| MVP | 静态手势识别 | 后端 MLP：数字 0–5、OK、点赞、张开手掌、握拳、剪刀等 |
| MVP | Web 实时可视化 | Canvas 叠加骨架、标签、置信度、FPS |
| 进阶 | 动态手势识别 | 后端 LSTM：挥手、左滑/右滑、放大/缩小、画圈 |
| 进阶 | 手势→网页动作映射 | 控制图片轮播 / 视频播放 / PPT 翻页 / 画布 |
| 进阶 | 在线数据采集页 | 网页录制样本、打标签、入库 |
| 进阶 | 后台模型管理 | 触发训练、查看指标、切换/热加载模型 |
| 扩展 | 多用户 / 会话隔离 | 每个 WebSocket 连接独立的序列缓冲与状态 |

## 2. 需求分析

### 2.1 功能性需求
- **FR-1 浏览器采集**：通过 `getUserMedia` 获取摄像头流，支持设备选择、分辨率、镜像。
- **FR-2 前端手部检测**：浏览器端 MediaPipe 输出关键点与边界框，支持单手/双手。
- **FR-3 关键点上行**：前端归一化后通过 WebSocket 发送 63 维特征（或原始关键点）。
- **FR-4 静态识别**：后端对单帧特征分类，回传标签与置信度。
- **FR-5 动态识别**：后端按连接维护滑窗，对连续 N 帧序列分类。
- **FR-6 结果下行与映射**：前端接收结果，渲染并触发网页内动作（可配置开关）。
- **FR-7 可视化**：Canvas 实时叠加关键点、骨架、标签、FPS、延迟。
- **FR-8 数据采集**：Web 采集页录制样本，经 REST 入库。
- **FR-9 模型训练与管理**：后端训练脚本/接口，输出可部署模型并支持热加载。

### 2.2 非功能性需求
- **性能**：端到端延迟 < 150ms；前端渲染 ≥ 24 FPS；关键点上行频率可降采样（如 15–20 Hz）。
- **隐私**：默认只传关键点，不传原始图像；HTTPS/WSS 加密传输。
- **鲁棒性**：一般室内光照、复杂背景、不同肤色稳定工作。
- **可用性**：手势触发需去抖/冷却，避免误触；断线自动重连。
- **可扩展性**：后端无状态化（状态随连接），支持多客户端并发。
- **可移植性**：前端纯 Web（Chrome/Edge/Firefox），后端 Python 跨平台/可容器化。

### 2.3 用例
```
访客 → 打开网页 → 授权摄像头 → 做手势 → 实时看到骨架+标签 → 触发网页动作(翻页/轮播)
教师 → 演示模式网页 → 手势左/右滑 → 幻灯片翻页
开发者 → 采集页 → 录制手势样本入库 → 后台触发训练 → 热加载新模型
```

## 3. 技术选型与论证

### 3.1 部署形态对比（为什么选 B/S + 前端 MediaPipe）

| 方案 | 关键点提取位置 | 网络传输内容 | 优点 | 缺点 | 是否采用 |
|------|----------------|--------------|------|------|----------|
| **A. 前端 MediaPipe + 后端分类** | 浏览器(WASM) | 仅 63 维关键点 | 延迟低、带宽极小、隐私好、后端轻、易并发 | 前端需加载 WASM 模型(~数MB) | ✅ **主方案** |
| B. 前端推帧 + 后端全流程 | 后端 | 视频帧(大) | 前端简单 | 带宽大、延迟高、后端算力压力大、隐私差 | ❌ 备选 |
| C. 纯前端(JS 端到端) | 浏览器 | 无(全本地) | 零后端延迟 | 难做训练管理/数据沉淀/复杂模型、不便展示后端 | ⚠️ 降级方案 |

**结论**：采用**方案 A**。它把"重计算的视频处理"留在客户端、"模型推理与数据/模型治理"放在后端，网络只传几百字节关键点，是 Web 实时手势识别的工程最优解，也最契合"系统设计与实现"的 B/S 架构叙事。方案 C 可作为离线/弱网降级（前端内置 TFLite 模型本地推理）。

### 3.2 算法路线对比（核心识别方案，与部署无关）

| 方案 | 原理 | 优点 | 缺点 | 是否采用 |
|------|------|------|------|----------|
| **MediaPipe 关键点 + 分类器** | 21 关键点 → 归一化 → 分类 | 轻量、实时、对背景/光照鲁棒、维度低 | 依赖关键点检测质量 | ✅ **主方案** |
| 端到端 CNN（原始图像） | 直接对图像分类 | 不需关键点 | 需大量数据、对背景敏感、算力高 | ❌ |
| 传统 CV（肤色+凸包） | HSV 分割+凸缺陷 | 可解释、无需训练 | 对光照/肤色极敏感 | ⚠️ 论文对比基线 |

### 3.3 技术栈

**前端**
| 类别 | 选型 | 用途 |
|------|------|------|
| 框架 | Vue 3 + Vite（或 React 18，任选） | SPA、组件化 |
| 手部检测 | `@mediapipe/tasks-vision`（HandLandmarker, WASM/WebGL） | 浏览器端 21 关键点 |
| 视频/绘制 | `getUserMedia` + `<video>` + `<canvas>` | 采集与骨架叠加 |
| 通信 | 原生 WebSocket（或 socket.io） | 关键点上行/结果下行 |
| UI | Element Plus / Ant Design Vue | 控件、布局 |
| 构建 | Vite | 打包、HMR |

**后端**
| 类别 | 选型 | 版本建议 | 用途 |
|------|------|----------|------|
| 语言 | Python | 3.9–3.11 | 主开发语言 |
| Web 框架 | FastAPI + Uvicorn | 0.110+ | REST + WebSocket |
| 静态分类 | TensorFlow-Keras（MLP）/ scikit-learn | 2.13+ | 单帧分类 |
| 动态分类 | TensorFlow-Keras（LSTM） | 2.13+ | 时序分类 |
| 数值/数据 | NumPy / pandas | — | 特征与数据集 |
| 数据库 | SQLite（起步）/ PostgreSQL | — | 样本/模型元数据 |
| 部署 | Docker + Nginx（反代 + WSS） | — | 容器化与 HTTPS |

> 说明：前端 MediaPipe 模型文件（`hand_landmarker.task`）与 WASM 运行时需托管为静态资源；首屏会下载数 MB，可做加载提示与缓存。

### 3.4 MediaPipe 手部关键点说明
浏览器端 `HandLandmarker` 输出 **21 个关键点**，每点含 `(x, y, z)` 归一化坐标（x,y ∈[0,1] 相对画面宽高，z 为相对手腕深度）。
```
0: 手腕(WRIST)
1-4: 拇指(THUMB)         5-8: 食指(INDEX)
9-12: 中指(MIDDLE)       13-16: 无名指(RING)
17-20: 小指(PINKY)
```
每根手指 4 个点（指根→指尖），加手腕共 21 点。这 63 维（21×3）特征即上行/分类的输入来源。

## 4. 系统总体架构

### 4.1 分层架构
```
┌──────────────────────────── 浏览器 (Client) ────────────────────────────┐
│  采集层   getUserMedia → <video>                                          │
│  感知层   MediaPipe HandLandmarker (WASM/WebGL) → 21 关键点               │
│  特征层   关键点归一化 → 63 维特征                                         │
│  可视化   <canvas> 叠加骨架/标签/FPS                                       │
│  动作层   手势 → 网页内动作（轮播/翻页/播放/画布）                         │
└───────────────┬───────────────────────────────▲──────────────────────────┘
                │ WebSocket 上行: 63维特征(JSON)   │ 下行: {label, conf, ts}
                ▼                                  │
┌──────────────────────────── 服务端 (FastAPI) ───────────────────────────┐
│  通信层   WebSocket 端点（每连接一个会话上下文）+ REST API                │
│  识别层   静态分类(MLP) │ 动态分类(LSTM) │ 后处理(阈值/投票/冷却)         │
│  序列层   按连接维护 30 帧滑窗缓冲                                         │
│  服务层   数据采集入库 │ 训练触发 │ 模型管理/热加载                        │
│  存储层   模型文件(.keras/.tflite) │ SQLite(样本&模型元数据) │ 数据集     │
└───────────────────────────────────────────────────────────────────────┘
```

### 4.2 实时数据流（运行时，B/S）
```
[浏览器]
摄像头帧 → MediaPipe 检测 → 21 关键点
  → 前端归一化（平移到手腕原点 + 尺度归一）→ 63 维特征
  → Canvas 绘制骨架（本地，立即）
  → WebSocket 发送 {mode, feature[63], ts}（按 15–20Hz 降采样上行）
        │
        ▼
[服务端]
接收特征
  ├─→ 静态分支：MLP → 标签+置信度
  └─→ 动态分支：入该连接滑窗(30帧) → 满则 LSTM → 动作标签
  → 后处理（置信度阈值 + 多帧投票 + 触发冷却）
  → 通过 WebSocket 回 {label, confidence, ts}
        │
        ▼
[浏览器]
接收结果 → 更新标签/置信度显示 → 触发网页动作映射（可开关）
```

### 4.3 离线训练流（开发/运维时）
```
Web采集页/采集脚本 → 样本入库(CSV/NPY + SQLite元数据) → 预处理(归一化/划分)
  → 训练(MLP / LSTM) → 评估(准确率/混淆矩阵) → 导出模型(.keras/.tflite/.pkl)
  → 后端模型管理接口热加载 → 识别层立即生效
```

### 4.4 关键设计决策
- **关键点提取放前端**：把最耗算力的视频处理留在客户端浏览器（WASM/GPU），后端只做轻量分类，单机可支撑大量并发；网络仅传 63 维特征（几百字节/帧）。
- **关键点归一化是准确率核心**：所有点平移使手腕为原点，再除以手掌基准长度（手腕→中指根距离）做尺度归一。识别**与手在画面中的位置、远近、大小无关**，极大提升泛化。可在前端做（减少上行歧义）或后端做（前端只传原始点），本方案默认**前端归一化**。
- **会话状态随连接**：动态手势的滑窗缓冲、冷却状态绑定到单个 WebSocket 连接，后端业务无全局可变状态，天然支持多用户并发与水平扩展。
- **静态与动态分离**：静态走单帧快路径，动态走滑窗序列，由前端 `mode` 字段或后端双路并行决定。
- **后处理去抖**：连续 K 帧一致才确认手势，触发后进入冷却期，避免抖动误触发。
- **降级方案**：弱网/离线时前端可切换内置 TFLite 模型本地推理（方案 C），保证可用性。

## 5. 模块详细设计

### 前端模块（`frontend/src/`）

#### 5.1 摄像头模块 `composables/useCamera.js`
- 职责：封装 `getUserMedia`，绑定到 `<video>`，支持设备枚举/切换、分辨率、镜像。
- 接口：`startCamera(deviceId?)`、`stopCamera()`、`listDevices()`。
- 要点：自拍镜像（CSS `transform: scaleX(-1)` 或 Canvas 翻转）让交互更直觉。

#### 5.2 手部检测模块 `composables/useHandLandmarker.js`
- 职责：加载 `HandLandmarker`（WASM + `hand_landmarker.task`），逐帧 `detectForVideo`。
- 接口：`init()`、`detect(video, timestamp) -> landmarks[]`。
- 参数：`numHands`、`minHandDetectionConfidence`、`minTrackingConfidence`、`runningMode:'VIDEO'`。

#### 5.3 特征与绘制 `utils/features.js` + `utils/draw.js`
- `normalizeLandmarks(landmarks) -> Float32Array(63)`：前端归一化（见 7.1）。
- `drawSkeleton(ctx, landmarks)`：Canvas 绘制 21 点与连线。

#### 5.4 通信模块 `composables/useGestureSocket.js`
- 职责：维护 WebSocket，发送特征、接收结果，断线自动重连、心跳。
- 接口：`connect()`、`send(feature, mode)`、`onResult(cb)`、`close()`。

#### 5.5 动作映射 `utils/actionMapper.js`
- 职责：将稳定手势映射到网页内动作（轮播翻页、视频播放/暂停、画布绘制等），可配置、可开关。
- 示例：`swipe_right→下一张`、`palm→暂停`、`thumbs_up→点赞`。

#### 5.6 页面/组件 `views/`
- `RecognizeView`：实时识别主页面（视频+Canvas+结果面板）。
- `CollectView`：数据采集页（选类别、录制、计数、提交）。
- `DemoView`：手势控制演示页（图片轮播/播放器）。
- `AdminView`：模型管理页（指标、训练触发、模型切换）。

### 后端模块（`backend/app/`）

#### 5.7 WebSocket 端点 `api/ws.py`
- 职责：`/ws/recognize` 接收特征帧，调用识别管线，回传结果；每连接持有独立 `SessionContext`（滑窗 + 冷却状态）。

#### 5.8 特征校验 `core/features.py`
- 职责：校验/补全上行特征维度；若前端传原始点则在此归一化。
- 接口：`validate(feature) -> np.ndarray(63,)`。

#### 5.9 静态识别 `core/static_classifier.py`
- 职责：加载 MLP 模型，对单帧特征预测标签+置信度。`predict(feature)->(label,conf)`。

#### 5.10 动态识别 `core/dynamic_classifier.py`
- 职责：对会话滑窗（30×63）满窗后送 LSTM。`update(ctx, feature)->Optional[(label,conf)]`。

#### 5.11 后处理 `core/postprocess.py`
- 职责：置信度阈值、多帧投票、触发冷却。`stabilize(ctx,label,conf)->Optional[label]`。

#### 5.12 REST 服务 `api/rest.py`
- `POST /api/samples`：采集样本入库；`GET /api/labels`：手势标签表；
- `POST /api/train`：触发训练（后台任务）；`GET /api/models`、`POST /api/models/activate`：模型管理与热加载；`GET /api/metrics`：最近评估指标。

#### 5.13 配置 `core/config.py`
- 集中管理：手势标签表、模型路径、阈值、滑窗长度、冷却时间、CORS/允许域、上行频率限制。

## 6. 数据集设计与采集

### 6.1 静态手势数据集
- **类别（建议 10 类起步）**：`fist(握拳)`、`palm(张开)`、`ok`、`thumbs_up(点赞)`、`peace(剪刀/胜利)`、`point(食指指向)`、`number_1~5` 等，可按需裁剪。
- **采集量**：每类 **≥ 500 帧**，多人（≥3人）、多角度、多远近、多光照采集，提升泛化。建议训练:验证:测试 = 7:1.5:1.5。
- **存储格式**：CSV，每行 = `label, x0,y0,z0, x1,y1,z1, ..., x20,y20,z20`（共 1+63 列）。

```
data/static/gesture_data.csv
label,x0,y0,z0,...,x20,y20,z20
ok,0.0,0.0,0.0,0.12,-0.05,0.01,...
fist,...
```

### 6.2 动态手势数据集
- **类别（建议 6 类）**：`swipe_left`、`swipe_right`、`swipe_up`、`swipe_down`、`zoom_in`、`zoom_out`（外加 `none/idle` 负类很重要）。
- **采集量**：每类 **≥ 80 段**，每段固定 `SEQ_LEN=30` 帧。
- **存储格式**：每段一个 `.npy`，形状 `(30, 63)`；或统一打包成 `(N, 30, 63)` + 标签数组。

```
data/dynamic/
  swipe_left/  seq_0001.npy ... (each (30,63))
  swipe_right/ ...
  none/        ...
```

### 6.3 在线采集（Web 采集页 `CollectView` + `POST /api/samples`）
- 网页选择类别 → 实时预览画面与关键点 → 点击/按住录制；
- **静态模式**：前端逐帧把归一化特征 POST/批量提交，后端追加到 CSV；
- **动态模式**：前端凑满 30 帧序列后提交一段，后端存为 npy；
- 页面显示各类样本计数、支持撤销最近一条，保证类别均衡。
- 兼容离线采集脚本 `scripts/collect_data.py`（OpenCV+MediaPipe，本地直接写数据集），便于无前端时批量采。

### 6.4 数据质量要点
- **类别均衡**：各类样本数尽量接近，否则模型偏向多数类。
- **负类必备**：动态识别必须有 `none` 类（无动作/随机移动），否则误触发严重。
- **多样性**：换人、换手、换背景、换光线。这是泛化的关键，远比堆同一个人的样本有用。
- **采集端一致性**：训练样本与线上推理都用**同一套归一化**（前端归一化），避免训练/推理特征分布不一致。

## 7. 算法与模型设计

### 7.1 关键点归一化（核心算法）
```
输入: 21 个点 (x_i, y_i, z_i)
1. 平移: 以手腕(点0)为原点 → x_i -= x_0, y_i -= y_0, z_i -= z_0
2. 尺度归一: base = 距离(点0, 点9)  # 手腕到中指根，作为手掌尺度
            所有坐标 /= base
3. 展平为 63 维向量
输出: 与位置、尺度无关的特征向量
```
> 该步骤让"近处大手"和"远处小手"做同一手势时特征几乎相同，是准确率从 ~80% 提升到 ~97% 的关键。

### 7.2 静态手势模型（MLP）
- **输入**：63 维归一化特征。
- **结构**：`Dense(128,relu) → Dropout(0.3) → Dense(64,relu) → Dropout(0.3) → Dense(num_classes, softmax)`。
- **损失/优化**：`categorical_crossentropy` + Adam(1e-3)。
- **训练**：batch=32，epochs≈50，EarlyStopping 监控 val_loss。
- **替代基线**：scikit-learn 的 SVM(RBF) 或 RandomForest，用于论文对比。

### 7.3 动态手势模型（LSTM）
- **输入**：`(SEQ_LEN=30, 63)` 序列。
- **结构**：`LSTM(64, return_sequences=True) → LSTM(128) → Dense(64,relu) → Dropout → Dense(num_classes, softmax)`。
- **替代**：1D-CNN 或 GRU；轻量场景可用 1D-CNN（更快）。
- **要点**：序列需做时间维度对齐（统一 30 帧，不足补零/重采样）。

### 7.4 后处理稳定化
- **置信度阈值**：`conf < 0.7` 视为未识别。
- **多帧投票**：维护最近 N=8 帧预测，众数 ≥ 6 才确认。
- **触发冷却**：一次动作触发后 `COOLDOWN=1.0s` 内不再触发同类，防连发。

### 7.5 评估指标
- 准确率(Accuracy)、精确率/召回率/F1（按类）、混淆矩阵、推理延迟(ms)、FPS。
- 论文中给出：MLP vs SVM vs 传统CV基线 的对比表 + 混淆矩阵热力图。
- Web 特有指标：端到端延迟（前端发出→收到结果）、上行带宽、并发连接数下的吞吐。

### 8.3 前端：归一化（与训练完全一致）（`features.js`）
```js
// landmarks: [{x,y,z} * 21] → Float32Array(63)
export function normalizeLandmarks(lm) {
  const wrist = lm[0];
  const mid = lm[9];                         // 中指根，作为尺度基准
  const base = Math.hypot(mid.x - wrist.x, mid.y - wrist.y, (mid.z ?? 0) - (wrist.z ?? 0)) || 1e-6;
  const out = new Float32Array(63);
  for (let i = 0; i < 21; i++) {
    out[i * 3]     = (lm[i].x - wrist.x) / base;
    out[i * 3 + 1] = (lm[i].y - wrist.y) / base;
    out[i * 3 + 2] = ((lm[i].z ?? 0) - (wrist.z ?? 0)) / base;
  }
  return out;
}
```

### 8.5 前端：WebSocket 封装（断线重连）（`useGestureSocket.js`）
```js
export function useGestureSocket(url = `wss://${location.host}/ws/recognize`) {
  let ws, cb = () => {}, retry = 0;
  function connect() {
    ws = new WebSocket(url);
    ws.onopen = () => { retry = 0; };
    ws.onmessage = (e) => cb(JSON.parse(e.data));
    ws.onclose = () => { setTimeout(connect, Math.min(1000 * 2 ** retry++, 8000)); };
  }
  connect();
  return {
    send: (msg) => ws?.readyState === 1 && ws.send(JSON.stringify(msg)),
    onResult: (fn) => { cb = fn; },
    close: () => ws?.close(),
  };
}
```

### 8.6 后端：FastAPI WebSocket 端点（`api/ws.py`）
```python
from fastapi import APIRouter, WebSocket, WebSocketDisconnect
import numpy as np, time
from app.core.static_classifier import StaticClassifier
from app.core.dynamic_classifier import DynamicClassifier, SessionContext
from app.core.postprocess import Stabilizer

router = APIRouter()
static_clf = StaticClassifier.load()      # 进程启动时加载一次
dynamic_clf = DynamicClassifier.load()

@router.websocket("/ws/recognize")
async def recognize(ws: WebSocket):
    await ws.accept()
    ctx = SessionContext(seq_len=30)       # 每连接独立滑窗
    stab = Stabilizer(threshold=0.7, vote_n=8, cooldown=1.0)
    try:
        while True:
            msg = await ws.receive_json()
            if msg.get("type") != "frame":
                continue
            feat = np.asarray(msg["feature"], dtype=np.float32)
            if feat.shape[0] != 63:
                continue
            if msg.get("mode") == "dynamic":
                out = dynamic_clf.update(ctx, feat)       # 满窗才返回
                label, conf = out if out else (None, 0.0)
            else:
                label, conf = static_clf.predict(feat)
            stable = stab.stabilize(label, conf)          # 阈值+投票+冷却
            await ws.send_json({"type": "result", "label": stable,
                                "confidence": float(conf), "ts": msg.get("ts")})
    except WebSocketDisconnect:
        pass
```

### 8.7 后端：静态分类器（`core/static_classifier.py`）
```python
import numpy as np, tensorflow as tf, json
from app.core.config import settings

class StaticClassifier:
    def __init__(self, model, labels):
        self.model, self.labels = model, labels
    @classmethod
    def load(cls):
        model = tf.keras.models.load_model(settings.STATIC_MODEL_PATH)
        labels = json.load(open(settings.STATIC_LABELS_PATH, encoding="utf-8"))
        return cls(model, labels)
    def predict(self, feature: np.ndarray):
        probs = self.model(feature[None, :], training=False).numpy()[0]
        idx = int(probs.argmax())
        return self.labels[idx], float(probs[idx])
```

### 8.8 后端：动态分类器 + 会话上下文（`core/dynamic_classifier.py`）
```python
from collections import deque
import numpy as np, tensorflow as tf, json
from app.core.config import settings

class SessionContext:
    def __init__(self, seq_len=30):
        self.buf = deque(maxlen=seq_len); self.seq_len = seq_len

class DynamicClassifier:
    def __init__(self, model, labels): self.model, self.labels = model, labels
    @classmethod
    def load(cls):
        model = tf.keras.models.load_model(settings.DYNAMIC_MODEL_PATH)
        labels = json.load(open(settings.DYNAMIC_LABELS_PATH, encoding="utf-8"))
        return cls(model, labels)
    def update(self, ctx: SessionContext, feature: np.ndarray):
        ctx.buf.append(feature)
        if len(ctx.buf) < ctx.seq_len:
            return None
        seq = np.stack(ctx.buf)[None, ...]      # (1, 30, 63)
        probs = self.model(seq, training=False).numpy()[0]
        idx = int(probs.argmax())
        return self.labels[idx], float(probs[idx])
```

### 8.9 后端：训练脚本（`scripts/train_static.py`，节选）
```python
import pandas as pd, numpy as np, tensorflow as tf, json
from sklearn.model_selection import train_test_split

df = pd.read_csv("data/static/gesture_data.csv")
labels = sorted(df["label"].unique()); l2i = {l: i for i, l in enumerate(labels)}
X = df.drop(columns=["label"]).values.astype("float32")
y = tf.keras.utils.to_categorical(df["label"].map(l2i), len(labels))
Xtr, Xte, ytr, yte = train_test_split(X, y, test_size=0.3, stratify=df["label"], random_state=42)

model = tf.keras.Sequential([
    tf.keras.layers.Input((63,)),
    tf.keras.layers.Dense(128, activation="relu"), tf.keras.layers.Dropout(0.3),
    tf.keras.layers.Dense(64, activation="relu"),  tf.keras.layers.Dropout(0.3),
    tf.keras.layers.Dense(len(labels), activation="softmax"),
])
model.compile(optimizer="adam", loss="categorical_crossentropy", metrics=["accuracy"])
model.fit(Xtr, ytr, validation_split=0.2, epochs=50, batch_size=32,
          callbacks=[tf.keras.callbacks.EarlyStopping(patience=8, restore_best_weights=True)])
print("test acc:", model.evaluate(Xte, yte)[1])
model.save("backend/app/models/static_mlp.keras")
json.dump(labels, open("backend/app/models/static_labels.json", "w"), ensure_ascii=False)
```

## 13. 执行计划（分周排期）

> 总周期建议 8 周（可按课程/毕设时间压缩或拉长）。每周给出目标、产出物、验收点。

### 第 1 周：环境与骨架
- 搭建前后端工程骨架（Vite + FastAPI），打通 `npm run dev` 与 `uvicorn`。
- 前端跑通 `getUserMedia` + MediaPipe HandLandmarker，Canvas 画出 21 点骨架。
- **产出**：能在浏览器看到实时手部骨架。**验收**：本地稳定 ≥24FPS 绘制。

### 第 2 周：通信链路 + 归一化
- 实现 WebSocket 上行/下行；后端先返回「回声」结果跑通链路。
- 前端实现 `normalizeLandmarks`，后端写同逻辑测试向量交叉校验一致。
- **产出**：前端发特征→后端收到→回包显示。**验收**：端到端延迟 <150ms。

### 第 3 周：数据采集
- 实现离线采集脚本 + Web 采集页，采集 10 类静态手势（每类≥500帧，≥3人）。

## 14. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 首屏 WASM/模型下载慢 | 体验差 | 自托管 + gzip/br 压缩 + 长缓存 + 加载进度提示 |
| HTTPS 限制致摄像头不可用 | 局域网无法演示 | 提前配自签/Let's Encrypt；答辩用 localhost 或可信证书 |
| 前端/后端归一化不一致 | 准确率骤降 | 交叉测试向量比对；统一在前端归一化，后端只做维度校验 |
| 动态手势误触发 | 控制乱跳 | 必加 none 负类 + 投票 + 冷却 + 提高阈值 |
| 不同摄像头/光照泛化差 | 实测掉点 | 多人多环境采集；必要时数据增强(轻微抖动/缩放) |
| 浏览器兼容(GPU delegate) | 部分机器卡 | 提供 CPU delegate 回退；降低 numHands/分辨率 |
| 高并发后端延迟 | 多人卡顿 | 推理批处理/限上行频率/多 worker；模型转 TFLite 提速 |
| 训练数据不足 | 过拟合 | 增大采集量、Dropout、EarlyStopping、交叉验证 |
